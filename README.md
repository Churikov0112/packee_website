# DosBrosDev - мультиязычный сайт-витрина игр (Astro + Tailwind + TypeScript)

## 1. Что такое Astro и почему выбран именно он

**Astro** это современный фреймворк для создания быстрых веб-сайтов, который хорошо подходит для контентных и преимущественно статических проектов.

Почему Astro выбран для этого сайта:

1. **Статическая генерация по умолчанию**  
   Проект не требует сложной серверной логики в рантайме: страницы игр, локализованные страницы и политика конфиденциальности могут быть собраны заранее.

2. **Высокая производительность**  
   На выходе получается легкий статический HTML/CSS/JS без лишней клиентской нагрузки.

3. **Удобная структура для контента**  
   В проекте много текстовых данных на разных языках, и Astro хорошо сочетается с файловым хранением контента (`src/data/**`).

4. **Рациональность для данного типа сайта**  
   Для витринного/информационного сайта использование более «тяжёлых» решений с постоянным серверным рендерингом было бы избыточным.

## 2. Общий стек технологий проекта

- **Astro** - фреймворк для построения маршрутов, компонентов и статической сборки сайта
- **Tailwind CSS** - утилитарная стилизация интерфейса
- **TypeScript** - типизация в библиотечных модулях и конфигурации, повышение надежности кода
- **JSON** (`src/data/**`) - основной источник контента (локализации, данные игр, политика конфиденциальности, контент главной страницы)
- **Firebase Hosting** - хостинг статических файлов в продакшене
- **PostCSS + Autoprefixer** - постобработка CSS и совместимость по браузерам

```text
│   │   ├── i18n/                    # словари интерфейса по языкам
│   │   ├── privacy/                 # тексты политики конфиденциальности

│   ├── layouts/
│   │   └── BaseLayout.astro         # базовый layout (meta, html-обвязка)
│   ├── lib/                         # утилиты и функции получения данных
│   │   ├── i18n.ts                  # локали, словари, роутинг локалей
│   │   ├── games.ts                 # загрузка/локализация данных игр
│   │   ├── privacy.ts               # загрузка политики конфиденциальности
│   │   └── site.ts                  # загрузка контента главной
│   ├── pages/
│   │   ├── index.astro              # редирект с / на /en/
│   │   └── [locale]/                # локализованные страницы
│   │       ├── index.astro          # главная страница языка
│   │       ├── privacy.astro        # политика конфиденциальности
│   │       └── games/[slug].astro   # страница конкретной игры
│   └── styles/
│       └── global.css               # глобальные стили
├── astro.config.mjs                 # конфигурация Astro (static output)
├── tailwind.config.cjs              # конфигурация Tailwind
├── postcss.config.cjs               # конфигурация PostCSS
├── tsconfig.json                    # конфигурация TypeScript
├── firebase.json                    # настройки Firebase Hosting
├── package.json                     # скрипты и зависимости
└── README.md
```

### Краткая логика архитектуры

- Контент отделён от компонентов: UI в `src/components`, данные в `src/data`
- Все маршруты генерируются статически на этапе `npm run build`
- Локализация реализована через JSON-файлы и утилиты из `src/lib/i18n.ts`
- Данные игр агрегируются из `src/data/games/*.json` в `src/lib/games.ts`

## 5. Как запустить тестовую сборку проекта на локальном компьютере

### Шаг 1. Установить зависимости

```bash
npm install
```

### Шаг 2. Запустить dev-режим

```bash
npm run dev
```

Что делает эта команда:

- запускает локальный сервер разработки Astro
- пересобирает проект при изменениях файлов
- удобна для повседневной разработки и проверки верстки/контента

### Шаг 3. Открыть сайт в браузере

Обычно Astro выводит локальный адрес вида:

- `http://localhost:4321`

## 6. Как запустить production-версию локально

### Шаг 1. Собрать проект

```bash
npm run build
```

Что делает `npm run build`:

- собирает проект в production-режиме
- генерирует статические файлы сайта
- сохраняет результат в папку **`dist/`**

### Шаг 2. Локально открыть собранную версию

```bash
npm run preview
```

Что делает `npm run preview`:

- поднимает локальный сервер, который обслуживает именно собранный `dist/`
- позволяет проверить поведение, близкое к реальному продакшену

### Разница между `dev` и `preview`

- `dev` - режим разработки (быстрая итерация, hot reload, инструменты разработки)
- `preview` - проверка уже собранной production-версии

## 7. Подробная инструкция по деплою на Firebase Hosting (попросил нейронку, не проверял)
### 7.1. Что нужно установить заранее

1. Node.js и npm
2. аккаунт Google/Firebase
3. Firebase CLI

Установка Firebase CLI:

```bash
npm install -g firebase-tools
```

Проверка установки:

```bash
firebase --version
```

### 7.2. Авторизация в Firebase

```bash
firebase login
```

Откроется окно браузера для входа в Google-аккаунт.

### 7.3. Инициализация Hosting (если проект ещё не инициализирован)

```bash
firebase init hosting
```

Рекомендуемые ответы мастера настройки:

- выбрать существующий Firebase-проект;
- `Public directory`: **`dist`**;
- `Configure as a single-page app`: **Yes** (в проекте используется rewrite на `index.html`);
- `Set up automatic builds and deploys with GitHub`: по необходимости.

> В текущем проекте файл `firebase.json` уже содержит нужную публикацию из `dist/` и правило rewrite.

### 7.4. Роль `firebase.json`

`firebase.json` определяет:

- какую папку публиковать (`hosting.public = "dist"`);
- какие файлы игнорировать;
- правила rewrite (все запросы направляются на `/index.html`, что важно для роутинга).

### 7.5. Сборка перед деплоем

Перед публикацией обязательно выполнить production-сборку:

```bash
npm run build
```

После этого в `dist/` окажутся актуальные файлы сайта, которые и будут отправлены в Hosting.

### 7.6. Деплой

```bash
firebase deploy
```

После успешного выполнения Firebase опубликует новую версию сайта.

## 8. Как добавить новую игру на сайт после того, как проект уже задеплоен на Firebase

### Шаг 1. Подготовить `slug` новой игры

1. Придумайте уникальный `slug` (например: `my-new-game-2026`)
2. Используйте только безопасные символы для URL:
   - латинские буквы в нижнем регистре
   - цифры
   - дефисы
3. Убедитесь, что такого `slug` ещё нет среди файлов в `src/data/games/`

### Шаг 2. Создать JSON-файл игры

1. Создайте файл:

```text
src/data/games/<slug>.json
```

2. Возьмите за шаблон любой существующий файл из `src/data/games/`
3. Заполните обязательные поля:
   - `slug` — должен совпадать с именем файла (без `.json`)
   - `icon` — путь к иконке игры в `public/images/games/<slug>/icon.webp`
   - `screenshots` — массив путей к скриншотам
   - `downloads` — массив платформ/ссылок (Google Play, RuStore, Drive и т.д.)
   - `order` — порядок отображения карточки на главной
   - `published` — `true`, если игра должна отображаться на сайте
   - `translations` — переводы для всех поддерживаемых языков

### Шаг 3. Заполнить переводы в `translations`

Для каждого языка нужно добавить объект вида:

- `title`
- `shortDescription`
- `description`
- `features` (массив строк)
- `seoTitle`
- `seoDescription`

Важно:

1. Не удаляйте языки, которые уже поддерживает сайт
2. Если перевод временно не готов, лучше всё равно добавить корректный временный текст, чтобы не было пустых блоков
3. Следите, чтобы `features` был именно массивом строк

### Шаг 4. Добавить изображения в `public`

1. Создайте папку:

```text
public/images/games/<slug>/
```

2. Положите туда:
   - `icon.webp` (иконка игры)
   - скриншоты (например `1.webp`, `2.webp`, `3.webp` ...)
3. Пути в JSON должны точно совпадать с реальными файлами

Пример:

- в JSON: `/images/games/my-new-game-2026/1.webp`
- в файловой системе: `public/images/games/my-new-game-2026/1.webp`

### Шаг 5. Запустить локальную проверку

После изменений обязательно выполните:

```bash
npm install
npm run check
npm run build
npm run preview
```

Что проверяем:

1. `npm run check` - нет ошибок по типам и Astro-диагностике
2. `npm run build` - новая игра попала в статическую сборку
3. `npm run preview` - страница игры открывается локально и корректно выглядит

### Шаг 7. Проверить маршруты новой игры

Откройте в браузере (локально через preview):

- `/<locale>/` - карточка новой игры должна быть в списке
- `/<locale>/games/<slug>` - страница игры должна открываться
- проверьте несколько языков, а не только один

### Шаг 8. Подготовить и отправить изменения в репозиторий

1. Зафиксируйте изменения в Git
2. Отправьте коммит в ветку/репозиторий, из которого вы обычно деплоите

### Шаг 9. Пересобрать проект перед деплоем

Даже если сайт уже давно задеплоен, перед новой публикацией нужно снова собрать актуальную версию:

```bash
npm run build
```

Именно содержимое свежей папки `dist/` будет отправлено на Firebase Hosting
